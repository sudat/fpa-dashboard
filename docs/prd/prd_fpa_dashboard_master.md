# 予実分析ダッシュボード 正本PRD

> このドキュメントは、2026-04-13 時点の壁打ち結果を反映した正本です。  
> 既存の `prd_fpa_dashboard.md` は、プロトタイプと初期構想を引き継ぐための履歴資料として扱います。

## 1. このドキュメントの目的

ログラスの月次データをもとに行っている予実分析業務を整理し、次に作るダッシュボードの ToBe を業務面とシステム面の両方から定義する。  
本ドキュメントは、今後の仕様確認、画面設計、実装判断の基準として使う。

## 2. 目指す状態

このプロダクトの主目的は、**予実分析会の質を上げること**である。  
単に資料作成を早くするだけではなく、会議の場で差異の原因を把握し、着地への影響を確認し、そのまま意思決定まで進められる状態を作る。

また、経営層と事業部が前向きに使うため、**モダンな SaaS のように軽く、見やすく、迷いにくい体験**を重視する。

## 3. スコープの前提

### 3.1 引き継ぐもの

- ログラス出力を起点に業務を回すこと
- 単月、YTD、着地見込の 3 つの時間軸で分析すること
- 全社と事業部の両方を見ること
- 科目体系は全社共通であること
- FPA が中間で論点整理、内容精査、示唆追加を担うこと

### 3.2 引きずらないもの

- プロトタイプの UI レイアウト
- プロトタイプの画面分割
- Vanilla JS や GAS HTML Service 前提の実装都合
- コメントの見せ方や入力方法の暫定仕様

プロトタイプは仕様確定版ではなく、業務理解のための試作とみなす。

## 4. 現状業務（AsIs）

### 4.1 現状の業務目的

ログラスから取得したデータをもとに、月次の予実比較、着地見込分析、事業部コメント収集を行い、予実分析会で経営層に報告している。

### 4.2 現状の分析軸

#### 単月度

- A: 前月に設定した対象月の見込
- B: 対象月の実績
- B-A
- C: 対象月の前年同月実績
- B-C

#### YTD

- A: 前月に設定した対象月見込を含めた累計
- B: 対象月までの実績累計
- B-A
- C: 対象月までの前年実績累計
- B-C

#### 着地見込

- A: 前月に設定した年度末までの見込
- B: 対象月までの実績 + 年度末までの見込
- B-A
- C: 前年実績
- B-C

### 4.3 現状の業務フロー

1. ログラスから xlsx ファイルを取得する
2. Google シートに貼り付ける
3. 関数で比較表を作る
4. Google Docs に転記する
5. Docs を Slack で展開する
6. 事業部が Docs にコメントを書く
7. FPA が内容を確認し、必要に応じて追加確認する
8. 予実分析会の前に経営層へ資料共有する
9. 予実分析会で経営報告する

### 4.4 現状データの制約

- 入力は xlsx 形式
- 1 ファイルで約 6 万行ある
- 利用する列は既存引継書に記載の 10 列を前提とする
- 科目と部署はかなり細かい粒度だが、個別取引を一意に特定するキーは持っていない

## 5. 現状課題

### 5.1 業務課題

- シート加工から Docs 転記までの手作業が重い
- コメント対象が曖昧で、FPA と事業部の往復が発生しやすい
- FPA が論点整理、精査、示唆追加を人力で全部担っている

### 5.2 会議課題

- Docs が静的なため、会議中に角度を変えて確認しづらい
- 「前月どうだったか」「内訳はどうか」「もっと細かく見たい」に即答しづらい
- グラフが弱く、推移や差異の分解をその場で示しづらい
- その結果、議論が浅くなり、「持ち帰って確認します」が発生しやすい

### 5.3 システム課題

- 新しい科目や部署が出たときのマスタ運用が重い
- 6 万行 xlsx を扱う前提で、取込・集計・表示の負荷を考える必要がある
- プロトタイプは単月と着地見込に寄っており、YTD や図表コメントの設計が不足している

## 6. ToBe（業務）

### 6.1 業務のゴール

予実分析会で、以下の順に議論を進められる状態を作る。

1. 差異の原因を把握する
2. 着地への影響を確認する
3. どうするか意思決定する

### 6.2 会議の基本導線

- 入口は全社の着地見込とする
- その後、同じ型の画面のまま YTD、単月へ切り替える
- 全社はトーン把握に使う
- 実際の議論は各事業部に降ろして進める
- 論点カードを先に並べるより、まず数字を速く確かめられることを優先する

### 6.3 コメント運用

- 事業部コメントは任意入力とする
- 総括コメントは残せるようにする
- 図表ごとのコメントも残せるようにする
- FPA は事業部コメントを確認し、必要に応じて追加確認を行う
- 経営層も図表ごとにコメントできるようにする
- 経営層コメントでは、問い、意思決定、宿題を残せるようにする

### 6.4 公開フロー

理想の流れは以下とする。

1. データアップロード
2. 比較表・図表の自動生成
3. FPA が数値整合性を確認
4. 事業部へ公開
5. 事業部が必要に応じてコメント
6. FPA が論点整理、精査、示唆追加
7. 経営層が確認し、会議に入る

「アップロードから 10 分後に Slack 自動送信すること」は必須ではない。  
必須なのは、**アップロード後に FPA 確認を経て、事業部がすぐ見始められる状態になること**である。

## 7. ToBe（システム）

### 7.1 体験方針

- 表を主役として残す
- 同じ論点を、表とグラフで素早く行き来できるようにする
- 一画面内で時間軸や分析角度を切り替えても迷わないようにする
- 重く見えず、直感的で、SaaS らしい手触りを出す

### 7.2 分析画面の基本方針

- 着地見込、YTD、単月は同じ型の UI で切り替える
- 全社と事業部を同じ文法で見られるようにする
- 科目体系は共通軸として扱う
- 重要科目セットは固定前提でよい

### 7.3 画面全体の構成

- 左サイドバーは薄くし、トップレベルは「予実分析」「管理」の 2 つに絞る
- 予実分析の主画面では、上部に「全社」「事業部タブ群」を横スクロールで並べる
- 時間軸の切替は、着地見込、YTD、単月の 3 つをピル型タブで扱う
- ファーストビューは、固定の主要科目だけを並べたサマリー行とする
- ファーストビューで最初に見える列は、着地見込、前月見込、前年差とする
- サマリーの下はタブ切替とし、推移グラフ、詳細表を同じ文脈の中で見られるようにする
- 図表コメントは右サイドに置き、閲覧文脈を崩さずに読めるようにする

### 7.4 推移グラフ

- 推移グラフはファーストビュー直下のデフォルトタブとする
- 横軸は月次時系列とする
- 予実分析対象月は縦のハイライト帯で強調する
- 初期表示の系列は「実績」「見込」の 2 本とする
- 縦軸は、金額表示と GMV 比率表示を切り替えられるようにする
- 金額 / GMV 比率の切替は、グラフ右上の小トグルで扱う
- 推移グラフは流れを見る場とし、グラフ上の月クリックでは別の画面状態へ自動遷移しない

### 7.5 差異分解グラフ

- 差異分解グラフは、推移の次に「どこが差異を作っているか」を掴むための補助ビューとする
- 分解軸は場面で切り替える
  - 全社タブでは事業部ごとに分解する
  - 事業部タブでは明細科目ごとに分解する
- グラフ型は横棒グラフとする
- 初期表示は差異影響の大きい上位 5 本前後とする
- 並び順は差異絶対値の大きい順とする
- 色分けは良化 / 悪化の 2 色に絞る
- 棒クリック時は、同じ画面内の詳細表で該当行へ寄せる
- その際は必要に応じて該当集計科目を一時展開し、自動スクロールし、行を数秒だけ薄くハイライトする

### 7.6 詳細表

- 詳細表はタブ切替で表示する
- 左端の主キーは、集計科目を起点にし、`+` 展開で明細科目を表示する
- 列形式は現行業務の文法をなるべく踏襲し、A / B / B-A / C / B-C の理解を崩さない
- 着地見込、YTD、単月の 3 つでも、列の並びはほぼ共通とする

### 7.7 ドリルダウン / ドリルスルー

- 集計科目を `+` 展開すると明細科目が見える
- 金額セルをクリックすると、明細科目 × 部署まで見える
- この粒度まで掘れれば、会議上は十分とする

### 7.8 比率表示

- 比率の標準分母は GMV 集計科目とする
- GMV は、ログラス内の細かい GMV 明細科目を集計した値を使う

### 7.9 コメント領域

- 図表コメントは右サイドに表示する
- コメントは個人名つきで時系列に並べる
- FPA、経営層、事業部で特別なコメントレイヤーは分けない
- 事業部コメントは冒頭だけ表示し、必要時に展開して全文を読む形とする
- コメントはタスク管理画面ではなく、今見ている図表の文脈の中で読むことを優先する
- 書いた本人は、自分のコメントを自由に更新できるようにする

### 7.10 科目選択時の連動方針

- 重要科目サマリーで選択した科目に応じて、自動で切り替わるのは推移グラフだけとする
- 詳細表やコメントまでは自動連動させず、ユーザーが明示的に見に行けるようにする

### 7.11 差異分解グラフから詳細表への連動

- 差異分解グラフから詳細表への遷移は、文脈を崩さない弱連動とする
- 詳細表タブへ寄る際は、該当行がどこかをすぐ見つけられることを優先する
- 強い画面遷移や別モーダルは使わず、同一画面内で読む流れを保つ

### 7.12 マスタ運用

- 新しい科目や部署は差分として検知する
- FPA は差分だけ承認・マッピングできるようにする
- 未マッピングがあっても、警告表示のうえで先に進めるようにする

### 7.13 取込後フロー

- 管理画面は最小限とし、アップロード後の確認も必要最低限に絞る
- FPA はまず取込結果ログを確認する
- 次に未マッピング警告を確認する
- その後は管理画面で完結させず、実際の予実分析画面で数値を確認する
- 管理画面から予実分析画面への導線は「分析画面で確認」ボタンとする

## 8. 確定した判断

本壁打ちで確定した内容を整理する。

- 最優先ゴールは予実分析会の質向上
- 主役の時間軸は着地見込
- 会議の確認順は、全社着地見込 → YTD → 単月 → 事業部詳細
- 表は残す
- ただし、表だけではなく推移と掘り下げのためにグラフが必要
- トップレベルのナビゲーションは「予実分析」「管理」の 2 つに絞る
- 全社と各事業部は横スクロールタブで並列に扱い、フォーマットは統一する
- 時間軸の切替はピル型タブで扱う
- ファーストビューは固定の主要科目サマリー行とする
- 推移グラフはデフォルト表示で、対象月は縦のハイライト帯で強調する
- 推移グラフでは金額 / GMV 比率を切り替えられるようにする
- 差異分解グラフは横棒グラフとし、全社では事業部ごと、事業部では明細科目ごとに分解する
- 差異分解グラフは上位 5 本前後を差異絶対値順に表示し、色は良化 / 悪化の 2 色に絞る
- 詳細表は集計科目 → 明細科目の展開型とし、列形式は現行業務をほぼ踏襲する
- コメントは総括と図表単位の両方を扱う
- 図表コメントは事業部任意入力
- コメントは個人名つきの時系列表示とし、誰でも自分のコメントを更新できる
- 科目選択で自動連動するのは推移グラフだけとする
- 差異分解グラフから詳細表へ寄るときは、一時展開、自動スクロール、数秒の薄いハイライトを使う
- 公開前レビューの最優先は数値整合性
- 会議直前の差し替えはほぼ起きない
- 管理画面は最小限とし、取込後は「分析画面で確認」へ進む

## 10. 次に確認すべきシステム仕様

次のフェーズでは、理想とするシステム仕様を以下の順で確認する。

1. **情報設計**  
   全社、事業部、時間軸、表、グラフ、コメントをどう配置するか。今回の骨格をもとにラベル、余白、優先順位を詰める

2. **分析体験**  
   時間軸切替、ドリルダウン、ドリルスルー、比率切替、表/グラフ切替をどう行うか。特に差異分解グラフと詳細表の連動を実装可能な形に落とす

3. **コメント体験**  
   総括コメント、図表コメント、経営層コメントの見せ方と、右サイドの情報密度をどう設計するか

4. **データ運用**  
   xlsx 6 万行前提での取込、集計、版管理、マスタ差分承認をどう実現するか

5. **モダン SaaS らしい UX 原則**  
   速い、軽い、迷わない、見やすい、触って理解できる、という体験をどのレベルまで求めるか
