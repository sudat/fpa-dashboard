# 予実管理 GAS Webアプリ — 引継ぎドキュメント

> このファイルは、プロトタイプと初期構想を引き継ぐための履歴資料です。  
> 現在の正本は `docs/prd/prd_fpa_dashboard_master.md` を参照してください。

## プロジェクト概要

ログラス（Loglass 経営管理）から取得したデータをもとに、月次の予実比較・着地見込分析・事業部コメント収集を一気通貫で行うWebアプリ。Google Apps Script（GAS）のHTML Serviceで構築し、Google Workspace内に閉じた形でデプロイする。

### 解決する課題

従来のワークフロー：ログラスからデータDL → Googleシートで比較表作成 → Google Docsに転記 → 事業部がコメント → 経営報告。このうち「シート加工→Docs転記」の手間と、Docsでは科目ドリルスルーができない（静的な表になる）という制約を解消する。

### 技術選定の経緯

- スクラッチ（Next.js等）は技術的には即座に構築可能だが、Workspace外にデータが出るためOAuth認証やセキュリティ審査の組織コストが重い
- Looker Studioはドリルスルーに制約があり、コメント入力ができない
- AppSheetは横並び比較表の再現にクセがある
- GAS WebアプリはWorkspace認証がそのまま効き、データがGoogle外に出ず、UIの自由度も十分 → 採用

## アーキテクチャ

### 全体構成

```
[ユーザーのブラウザ]
   ↓ Workspace認証
[GAS Webアプリ (HTML Service)]
   ├── フロントエンド: index.html（Tailwind CSS + Vanilla JS SPA）
   └── バックエンド: Code.gs（SpreadsheetApp で読み書き）
          ↓
[Google Sheets: 予実管理データ.gsheet]
   ├── ファクト（全数値データ）
   ├── コメント（事業部×月度）
   ├── マスタ_科目 / マスタ_事業部 / マスタ_ユーザー
   └── アップロード履歴
          ↓
[Google Drive: uploads/ フォルダ]
   └── アップロード原本のアーカイブ（タイムスタンプ付き）
```

### デプロイ方式

- コンテナバインド型：スプレッドシートに紐づいたApps Scriptプロジェクト
- 「次のユーザーとして実行」→ 管理者アカウント（全ユーザーの操作が管理者権限でSheetsにアクセス）
- 「アクセスできるユーザー」→ 組織内の全ユーザー
- アプリレベルの認可はマスタ_ユーザータブのロールで制御

### ファイル構成

- Code.gs（Code_v2.gs）— バックエンド。ログラスCSV対応版
- index.html — フロントエンド。SheetJS導入済みでxlsx/csv両対応

### Driveフォルダ構成

```
📁 FPA共有ドライブ/予実管理/
  📊 予実管理データ.gsheet    ← データストア兼Apps Scriptホスト
  📁 uploads/                 ← CSV/xlsx原本のアーカイブ先
```

## データモデル

### ログラスCSVのカラム構成（入力）

| 列 | 内容 | 備考 |
|----|------|------|
| A | 計画・実績 | シナリオ名。「実績」or「26年3月期着地見込0324時点」等。見込は毎月名前が変わる |
| B | 年月 | yyyy/mm/dd形式。Sheetsが勝手にDate型に変換する場合あり |
| C | ログラス科目コード | |
| D | 外部システム科目コード | 会計システム側のコード |
| E | 科目 | ログラスでの科目名称 |
| F | 科目タイプ | 「収益」or「費用」のみ |
| G | ログラス部署コード | |
| H | 外部システム部署コード | |
| I | 部署 | ログラスでの部署名称 |
| J | 金額 | 貸借区分なし。収益も費用もプラス、戻しのみマイナス |

### ファクトタブ（内部データストア）

ログラスCSVの10列 + 取込日時の11列構成。年月はyyyy-MMに正規化して保存。

| シナリオ | 年月度 | 科目コード | 外部科目コード | 科目 | 科目タイプ | 部署コード | 外部部署コード | 部署 | 金額 | 取込日時 |

### コメントタブ

事業部×月度の粒度（科目単位ではない）。改行込みの長文を格納。各事業部のトップページに表示される総括コメントの位置づけ。

| 年月度 | 部署 | コメント | ステータス | 更新者 | 更新日時 |

ステータスは「未入力」「入力済」「確認済」の3値。

### マスタ_科目

CSVアップロード時に`_syncMasters()`で自動登録される。表示順と集約グループは手動で設定。

| 科目コード | 科目名 | 表示順 | 集約グループ | 科目タイプ |

### マスタ_事業部

同じくCSVアップロード時に自動登録。CSVに含まれない部署（コーポレート等）は手動追加。

| 部署名 | 部署コード | 外部部署コード |

### マスタ_ユーザー

手動設定が必須（アプリの認可の根拠）。未登録ユーザーは自動的に「閲覧」ロール。

| メールアドレス | ロール | 所属部署 | 氏名 |

ロール: 管理者 / 事業部 / 閲覧

### アップロード履歴

| ファイル名 | シナリオ | 年月度 | 取込日時 | 取込者 | 行数 | 削除行数 |

## 画面構成（5メニュー）

### ① ダッシュボード

- 対象月度と見込シナリオをセレクターで選択
- 主要科目をスコアカードで一覧（実績値 + vs見込差異 + vs前年差異）
- カードクリックで②単月度分析に遷移（該当科目が展開された状態）

### ② 単月度分析

- 縦軸: 勘定科目 / 横軸: 実績 | 見込 | 差異 | 差異率 | 前年実績 | 前年差異 | 前年差異率
- 科目行クリックでアコーディオン展開（部署別ドリルスルー）
- ドリルスルーは取得済みデータの絞り込みで実現（追加APIコール不要）

### ③ 着地見込分析

- ②と同じテーブル構造。基準シナリオセレクターが追加される
- 当月見込 vs 前月見込 vs 前年実績

### ④ コメント管理

- 事業部ごとにカード表示、textarea で長文入力
- 事業部担当者は自部署のみ編集可能。管理者は全部署を編集可能
- 管理者のみ「確認済にする」ボタンが表示される
- 閲覧者は全カードがdisabled

### ⑤ データ管理（管理者のみ表示）

- xlsx/csvをドラッグ&ドロップでアップロード
- プレビュー（シナリオ・年月度・部署の一覧 + 先頭20行）表示後に確定
- 洗い替え方式: 同一シナリオ×年月度のデータはDELETE→INSERT
- アップロード履歴のテーブル表示

## ユーザーストーリー

### データ管理者（経営企画部門）

1. ログラスからxlsxダウンロード（実績、見込、着地見込等）
2. ⑤データ管理でxlsxをアップロード → プレビュー確認 → 確定
3. ①ダッシュボードで全体感を確認、異常値のある科目を発見
4. ②単月度分析で該当科目をドリルスルーし、乖離の大きい部署を特定
5. ④コメント管理でコメント入力状況を確認、未入力の事業部にSlackで依頼
6. 全事業部のコメントが揃ったら「確認済」に変更
7. ①ダッシュボードのURLを経営会議のアジェンダに貼って報告完了（Docs作成不要）

### 事業部門担当者

1. 経営企画からURLとコメント依頼が来る
2. ①②で自事業部の数字を確認、ドリルスルーで乖離原因を特定
3. ④コメント管理で自部署のカードにコメント入力・保存
4. 必要に応じて③着地見込も確認・コメント

### 閲覧担当者（経営層）

1. 経営会議アジェンダのURLからアプリを開く（⑤は非表示）
2. ①で主要指標を一覧、②③でドリルスルー確認
3. ④で各事業部のコメントを閲覧

## 設計上の重要事項

### シナリオ名の扱い

ログラスの見込シナリオ名は毎月変わる（例: 「26年3月期着地見込0324時点」「26年3月期着地見込0424時点」）。フロントのセレクターはファクトタブから動的にdistinctして表示するので名前が変わっても問題ない。ただし前月見込と当月見込の比較時にはユーザーが正しいシナリオを手動選択する必要がある。

### 日付の正規化

ログラスのB列はyyyy/mm/dd。SheetsがDate型に変換する場合もある。`_normalizePeriod()`でDate型・文字列の両方をハンドリングし、内部的にはyyyy-MM文字列に統一。

### 洗い替えロジック

同一シナリオ×年月度の組み合わせで再アップロードすると、ファクトタブの該当行を全削除→新規追記。アップロード確定時に「既存データがN件あります」とUI上表示される。Drive側のuploadsフォルダにタイムスタンプ付きで原本が残るので巻き戻し可能。

実装はdeleteRowループではなく、全データ配列読み込み→フィルタ→clearContents→一括setValues方式（高速）。

### マスタ自動同期

CSVアップロード時に`_syncMasters()`が走り、未登録の科目・部署をマスタに自動追加。表示順・集約グループは空で入るので管理者があとから手動設定する運用。

### xlsx対応

フロントでSheetJS（CDN）を使ってxlsxをブラウザ上でCSVテキストに変換してからバックエンドに送る。バックエンド側のパース処理（Utilities.parseCsv）は一切変更なし。複数シートがある場合は1枚目だけ読む。

### 認証・認可

- 認証: GASデプロイ設定の「組織内の全ユーザー」でWorkspace認証
- 認可: マスタ_ユーザーのロール（管理者/事業部/閲覧）で制御
- `Session.getActiveUser().getEmail()`でログインユーザーを特定
- 未登録ユーザーは閲覧ロールにフォールバック
- 「次のユーザーとして実行」= 管理者 → エンドユーザーにSheetsの共有権限を付与する必要なし

### GASの制約・注意点

- スクリプト実行時間の上限: 6分（月次データ量なら問題なし）
- コード修正後は「デプロイ管理 → 新バージョン」が必要（忘れると古いコードのまま）
- UI描画速度: Sheets読み込みに数秒かかる場合あり。データ量が増えたらサマリキャッシュタブの検討を
- Sheetsの上限: 1000万セル。科目100×部署10×月12×シナリオ5 = 6万行程度なので余裕あり

### UIデザイン

- shadcn/uiのデザイントークン（CSS変数、border-radius、色体系）をVanilla CSSに移植
- フォント: Noto Sans JP（本文）+ JetBrains Mono（数値）
- 数値は等幅フォント・右揃え、差異はプラス=緑/マイナス=赤のバッジ
- SPA構成: サイドバー + コンテンツ領域のタブ切り替え

## セットアップ手順（実施済み）

1. FPA共有ドライブに「予実管理データ」スプレッドシートを作成
2. 6つのタブ（ファクト、コメント、マスタ_科目、マスタ_事業部、マスタ_ユーザー、アップロード履歴）をヘッダー付きで作成
3. 同じ場所に「uploads」フォルダを作成
4. スプレッドシートから「拡張機能 > Apps Script」でエディタを開く
5. コード.gsの中身をCode_v2.gsに差し替え、UPLOAD_FOLDER_IDを設定
6. HTMLファイル「index」を追加し、index.htmlの内容を貼り付け
7. Webアプリとしてデプロイ（組織内ユーザー向け）
8. マスタ_ユーザーに管理者を登録
